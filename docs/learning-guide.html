1. Project Overview & Motivation

The Problem

When an organization has 10+ teams using LLMs independently:

  • No cost visibility — $50K/month bill with no per-team breakdown
  • No access control — Interns calling GPT-4 Turbo ($30/1M output) for tasks GPT-4o-mini ($0.60/1M) handles
  • No rate protection — One runaway script exhausts the org's API quota
  • Vendor lock-in — Every team imports openai directly; switching to Claude touches every codebase
  • No audit trail — No record of what was sent, by whom, or what it cost

The Solution

A centralized API gateway that sits between teams and LLM providers. Teams send requests using the standard OpenAI-compatible API format. The gateway handles routing, cost tracking, access control, caching, and auditing — without changing how teams write code. Just change base_url and the API key.

3. Request Pipeline Deep Dive

Every request to /v1/chat/completions passes through this 8-step pipeline:

Step 1: Authentication

What: Extract Authorization: Bearer gw-xxx. SHA256 hash the token. Look up in api_keys table. Join with teams to get config.

key_hash = hashlib.sha256(bearer_token.encode()).hexdigest()
api_key = await db.execute(
    select(ApiKey).where(ApiKey.key_hash == key_hash, ApiKey.is_active == True)
)
team = api_key.team  # eager-loaded relationship

Why SHA256? We never store raw keys. If the DB leaks, attacker gets hashes, not usable keys. SHA256 is fast enough for per-request lookup but irreversible. Same approach as Stripe, GitHub, AWS.

Tradeoff: SHA256 vs bcrypt — API keys are 64+ random chars (high entropy), so brute-force is infeasible regardless. bcrypt's intentional slowness would add ~100ms per request, unacceptable for a proxy.

Step 2: Rate Limiting

What: Redis sorted set sliding window (60s). Count requests in window. If count ≥ team's rate_limit_rpm, return 429.

ZREMRANGEBYSCORE rate_limit:{team_id} -inf (now-60)   # Prune old
ZCARD rate_limit:{team_id}                              # Count current
ZADD rate_limit:{team_id} {timestamp} {request_id}     # Record new
EXPIRE rate_limit:{team_id} 120                         # Safety TTL

Why sliding window? Fixed windows have a burst problem: 59 requests at 0:59 + 59 at 1:01 = 118 in 2s. Sliding window looks at last 60s from now, accurately tracking actual rate.

Fail-open: If Redis is down, rate limiter returns allowed=True. Better to serve without rate limiting than block all traffic.

Step 3: Policy Evaluation

What: Fetch active policies for team + global (team_id IS NULL). Filter by fnmatch glob pattern against model name. Sort by priority desc. First match wins. Default: ALLOW.

# Example: Block GPT-4 for intern team
Policy(team_id="intern-team", policy_type="deny", resource="gpt-4*", priority=100)

Why glob? gpt-4* matches gpt-4o, gpt-4o-mini, gpt-4-turbo. No need to update policies when new model versions launch.

Step 4: Cache Check

What: Hash model + messages + temperature + top_p + max_tokens with SHA256. Check Redis. Only for temperature=0.

Why only temp=0? Deterministic output — same prompt always produces same response. At temp > 0, output is random; caching would return stale/wrong results.

Step 5: Provider Routing

What: Find providers supporting the model, sorted by preference. Try each with failover.

OpenAI models:  openai → azure → (no bedrock)
Claude models:  bedrock → (failover to next if down)

Steps 6-8: Cost → Audit → Usage

Cost: Look up model in pricing table, multiply tokens × rate per 1M. Prefix matching fallback for versioned names. Audit: Insert row with request_id, team, model, tokens, cost, latency, status, truncated bodies. Usage: Upsert daily aggregation row (team + model + provider + date).

6. Policy Engine

{
  "name": "Block expensive models for interns",
  "team_id": "intern-team-uuid",     // null = global
  "policy_type": "deny",              // "allow" or "deny"
  "resource": "gpt-4*",               // glob pattern
  "priority": 100,                    // higher wins
  "conditions": {
    "max_tokens": 4096,               // deny large requests
    "time_window": {"start_hour": 0, "end_hour": 8},
    "denied_models": ["gpt-4-turbo"]
  }
}

Evaluation Algorithm

  1. Fetch active policies for team + global
  2. Filter: fnmatch(resource, model)
  3. Sort by priority DESC
  4. First match wins (allow or deny)
  5. No match → default ALLOW

Why default allow? Principle of least surprise. If you haven't configured policies, everything works. You add restrictions incrementally.

7. Rate Limiting

Sliding Window vs Fixed Window vs Token Bucket

AlgorithmBurst ProblemMemoryAccuracyWe Use
Fixed WindowYes (boundary burst)O(1)Low
Sliding Window LogNoO(N) per teamHighYes
Token BucketControlled burstO(1)Medium
Leaky BucketNo burst allowedO(1)High

Why sliding window? It's the most accurate for "60 requests per minute" semantics. Token bucket allows bursts (which we don't want for LLM APIs that have their own rate limits). The O(N) memory cost is negligible — at 60 RPM, each team's sorted set has at most 60 entries.

Redis implementation: Sorted set where score = timestamp. ZREMRANGEBYSCORE prunes old entries in O(log N + M). All 4 operations run in a pipeline (single round-trip). 120s TTL is safety net for abandoned keys.

8. Response Caching

Cache Key

Input:  {model, messages, temperature, top_p, max_tokens, n}
        ↓ json.dumps(sort_keys=True)
        ↓ SHA256
Output: "cache:chat:a1b2c3d4..."  (64 hex chars)

Rules

  • Only cache when temperature = 0 — deterministic output, safe to reuse
  • TTL from config — default 3600s (1 hour)
  • Cache key includes all params — different max_tokens = different cache entry

Cache Invalidation

Pattern-based via SCAN + DELETE. Current limitation: model name not in key prefix (it's inside the hash), so you can only invalidate all entries. Improvement: prefix key with model name for targeted invalidation.

Key Design Choices

  • UUID PKs — No sequential IDs revealing record counts. Generated client-side, no DB round-trip.
  • key_hash not key — Raw key shown once on creation. Stored as SHA256. key_prefix (gw-abc1234) for UI identification.
  • JSONB conditions — Flexible policy conditions without migrations for each new condition type.
  • Unique constraint on usage_summaries — (team, model, provider, date) enables safe upsert for aggregation.
  • Body truncation — request/response bodies truncated to 500 chars in audit. Reduces PII exposure and table bloat.

1. Admin creates team:    POST /v1/teams {"name": "engineering"}
2. Admin generates key:   POST /v1/teams/{id}/keys {"name": "prod-key"}
   → Returns: {"key": "gw-abc123..."}    (shown ONCE)
   → Stored:  key_hash = SHA256("gw-abc123..."), key_prefix = "gw-abc1234"

3. Client uses key:       Authorization: Bearer gw-abc123...
   → Gateway: SHA256 → lookup api_keys → join team → proceed

Prefix gw- — Like sk- for OpenAI. Enables automated secret scanning (GitHub, GitGuardian). Show-once — Like AWS/Stripe. Forces secure storage. Lost key = revoke and regenerate.

Implementation: Add a middleware that generates request_id (UUID) and injects it into Python's logging context via contextvars. Every log call in the request lifecycle includes this ID for end-to-end tracing.

# middleware/request_id.py
import uuid
from contextvars import ContextVar
from starlette.middleware.base import BaseHTTPMiddleware

request_id_var: ContextVar[str] = ContextVar("request_id", default="")

class RequestIDMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        rid = str(uuid.uuid4())[:8]
        request_id_var.set(rid)
        response = await call_next(request)
        response.headers["X-Request-ID"] = rid
        return response

14. Failure Scenarios & Edge Cases

Scenario 1: Redis Down

What happens:
Rate limiter returns allowed=True (fail-open). Cache returns None (miss). All requests hit providers directly.
Problem:
No rate protection. No caching savings. Provider bills spike. A runaway script could exhaust quotas.
Fix:
In-memory fallback rate limiter (per-instance, not global). Redis Sentinel for automatic failover. Alert on Redis health check failure.

Scenario 2: PostgreSQL Slow/Down

What happens:
Auth lookup fails (can't verify API key). Audit log writes fail. Usage aggregation fails.
Problem:
All requests fail with 500. Complete outage even though LLM providers are healthy.
Fix:
(1) Cache valid API keys in Redis with 5-min TTL — auth works even if DB is down. (2) Queue audit writes to Redis list, drain to Postgres via background worker. (3) Circuit breaker on DB calls.

Scenario 3: All Providers Fail

What happens:
Each provider tried in order. All throw exceptions. Gateway returns RuntimeError("All providers failed").
Problem:
Client gets a 500 with no useful error. No way to distinguish "all down" from "your request is bad."
Fix:
Return 503 Service Unavailable with Retry-After header. Circuit breaker per provider — if a provider fails 5x in 10s, skip it for 30s. Return cached response if available (even for temp > 0) as degraded fallback.

Scenario 4: Provider Timeout / Partial Response

What happens:
httpx times out at 30s. Bedrock boto3 hangs. Partial JSON response can't be parsed.
Problem:
Provider may have consumed tokens (they charge on generation start). We don't log the cost because the response failed to parse.
Fix:
Log partial requests with status="timeout" and estimated cost. Add request timeout to httpx client. For streaming, track incremental token count.

Scenario 5: Gateway Crash Mid-Request

What happens:
LLM response received but gateway crashes before audit write.
Problem:
Cost shows up in provider invoice but not in our tracking. Usage summaries are under-counted.
Fix:
Write "request started" entry (status=pending) before calling provider. Update to completed/failed after. Pending entries older than 60s = lost requests. Reconcile with provider billing API.

Key Patterns

Circuit Breaker

States: CLOSED → OPEN → HALF_OPEN

CLOSED:  Normal operation. Track failures.
         If failures > threshold in window:
         → switch to OPEN

OPEN:    All requests fail-fast (no attempt).
         After timeout (30s):
         → switch to HALF_OPEN

HALF_OPEN: Allow 1 test request.
           If success: → CLOSED
           If failure: → OPEN

Exponential Backoff

attempt = 0
while attempt < max_retries:
    try:
        return await provider.call(request)
    except RetryableError:
        wait = min(base * 2^attempt, max_wait)
        wait += random.uniform(0, jitter)
        await asyncio.sleep(wait)
        attempt += 1

# base=0.5s, max_wait=30s, jitter=0.5s
# Waits: 0.5s, 1s, 2s, 4s, 8s, 16s, 30s

Idempotency Keys

Client sends X-Idempotency-Key: abc123. Gateway checks Redis for this key. If found, return the stored response. If not, process request and store response with TTL. Prevents duplicate charges on network retries.

Database Optimizations

  • Audit log partitioning: Range partition by created_at (monthly). Old partitions detach and archive to S3. Queries with date filters only scan relevant partitions.
  • Read replica: Dashboard queries (usage, costs, top models) hit the replica. Writes (audit, usage upsert) go to primary.
  • Connection pooling: PgBouncer in front of PostgreSQL. Each gateway instance gets a bounded pool instead of unbounded connections.
  • Indexes: audit_logs(created_at, team_id), usage_summaries(date, team_id), api_keys(key_hash)

16. Security Deep Dive

Current Security Measures

  1. API key hashing — SHA256, shown once, prefix for ID
  2. Admin isolation — Separate admin API key for management
  3. Team isolation — Each key belongs to one team, no cross-team access
  4. Body truncation — Audit logs truncate to 500 chars, reducing PII exposure
  5. Env-based secrets — No credentials in code
  6. Docker network isolation — Postgres/Redis not exposed to host in production

Production Improvements

API Key Rotation

Allow teams to have multiple active keys simultaneously. New key created, old key set to expire in 24h. Client migrates to new key. Zero-downtime rotation.

# Rotation flow:
1. POST /v1/teams/{id}/keys {"name": "prod-key-v2"}  # new key created
2. Deploy new key to client applications
3. DELETE /v1/teams/{id}/keys/{old_key_id}            # revoke old key

Per-Key Rate Limiting

Currently rate limiting is per-team. A single runaway key in a team affects all keys. Fix: add rate_limit_rpm to api_keys table, check both team-level and key-level limits.

PII Redaction in Audit Logs

Before storing request/response bodies, scan for PII patterns (emails, SSNs, phone numbers) and replace with [REDACTED]. Use regex patterns for common formats. For production: integrate with a dedicated PII detection service.

17. Advanced Features

1. Smart Model Routing

What: Automatically choose the cheapest model that meets quality requirements.

Why: Teams often default to GPT-4o when GPT-4o-mini would suffice. A 16x cost difference at scale.

How:

# Policy condition: "auto_downgrade": true
# If request doesn't specify a model preference:
1. Estimate prompt complexity (token count, presence of code/math)
2. If simple (<500 tokens, no code): route to gpt-4o-mini
3. If complex: route to gpt-4o
4. Track quality feedback (thumbs up/down from teams) to tune thresholds

2. Hard Budget Enforcement

What: Reject requests when a team exceeds their monthly token budget.

Why: Currently budget is monitoring-only. Finance needs hard limits.

How:

# In gateway pipeline, between auth and rate limit:
budget_cache_key = f"budget:{team_id}:{current_month}"
spent = await redis.get(budget_cache_key)
if spent is None:
    spent = await db.query(SUM(usage_summaries.total_cost) WHERE month=current)
    await redis.setex(budget_cache_key, 60, spent)  # cache 60s
if spent >= team.token_budget_monthly:
    return 429, "Monthly budget exceeded"

Tradeoff: 60s cache means a team can exceed budget by up to 60s worth of requests. Acceptable for cost management; use shorter TTL if tighter control needed.

18. Design Decisions & Tradeoffs

Every major component had alternatives considered. This section explains what we chose, what we didn't, and why.

Rate Limiting: Sliding Window Log vs Token Bucket

Sliding Window (Chosen)Token Bucket
AccuracyExact count in windowAllows bursts up to bucket size
MemoryO(N) per team (N = requests in window)O(1) per team

Why chosenLLM APIs have their own strict rate limits. Allowing bursts at our gateway means hitting provider limits, which causes cascading failures. Strict enforcement at our layer prevents this.

SharedAll instances share one cachePer-instance, duplicated
PersistenceSurvives restartLost on restart
Latency~1ms network hop~0.01ms (in-process)
Why chosenWith multiple gateway instances, in-memory cache has low hit rate (request A cached on instance 1, request B goes to instance 2 = miss). Redis gives global cache. The 1ms overhead is negligible vs 500ms-5s LLM response.

Database: PostgreSQL vs MongoDB

PostgreSQL (Chosen)MongoDB
SchemaStrict schema + JSONB flexFully schemaless
TransactionsFull ACIDMulti-doc transactions (slower)
AggregationSQL aggregates, window functionsAggregation pipeline
Why chosenOur data is relational (teams → keys, teams → policies, teams → usage). SQL joins, unique constraints, and upserts are natural. JSONB gives schema-less flexibility for policy conditions. Async driver (asyncpg) is the fastest Python DB driver.

Auth: SHA256 vs bcrypt vs HMAC

SHA256 (Chosen)bcryptHMAC
Speed~0.001ms~100ms~0.001ms
PurposeFast irreversible hashIntentionally slowKeyed hash
Why chosenAPI keys are 64+ random chars = high entropy. Brute-force infeasible. bcrypt's slowness is needed for low-entropy passwords, not for random keys. HMAC needs a server-side secret (another thing to manage). SHA256 = fast + secure for this use case.

Caching Only at temperature=0

Alternative: Cache all responses with a "freshness" header.

Why not: At temperature > 0, the user explicitly wants randomness. Returning cached output defeats this. We'd need a "stale but fast" opt-in mechanism, which adds API complexity. The simple rule (temp=0 = cacheable) is easy to reason about and never returns unexpected results.

Fail-Open vs Fail-Closed

Chosen: Fail-open for rate limiting and caching. Fail-closed for auth (no DB = no auth = deny).

Why: Auth is a security boundary — you can't bypass it. Rate limiting and caching are optimizations — the core functionality (routing LLM requests) works without them. Blocking all traffic because Redis is down is worse than allowing some extra traffic.

20. Elevator Pitches

2-Minute Version

I built an AI Gateway — a centralized proxy that sits between an organization's engineering teams and LLM providers like OpenAI and AWS Bedrock. The problem it solves: when you have 10+ teams independently calling AI APIs, you get a $50K monthly bill with no visibility into who's spending what, no access control, and no rate protection.

The gateway exposes an OpenAI-compatible API, so teams only change their base URL. Every request goes through an 8-step pipeline: authentication, rate limiting, policy evaluation, caching, provider routing with failover, cost tracking, audit logging, and usage aggregation. Built with FastAPI, PostgreSQL, Redis, and a React dashboard for cost monitoring. The architecture is async end-to-end and horizontally scalable.

5-Minute Deep Version

Problem: At CVS Health, we had multiple teams calling OpenAI independently. The finance team came to us asking who spent $47K on GPT-4 last month. Nobody knew. No audit trail, no access control, one intern's test script was calling GPT-4 Turbo in a loop.

Solution: I designed and built a centralized AI Gateway. It's a FastAPI application that acts as a proxy between internal teams and multiple LLM providers. The key design decision was making it OpenAI-compatible — teams don't change their code, just the base URL and API key.

Technical depth: The request pipeline is the core. Auth uses SHA256-hashed API keys (like Stripe). Rate limiting uses a Redis sorted set with a sliding window — I chose this over a token bucket because LLM APIs have strict rate limits, and allowing bursts at our gateway means hitting their limits. The policy engine uses glob-pattern matching so gpt-4* covers all GPT-4 variants without updating policies when new versions launch. Caching only applies to temperature=0 requests (deterministic output). Provider routing has automatic failover — if OpenAI is down, it tries Azure OpenAI.

Cost tracking uses per-model token pricing with prefix matching for versioned model names. Usage is pre-aggregated into daily summaries (team + model + provider + date) so the dashboard reads ~1,000 rows instead of scanning millions of audit logs.

Scale considerations: The gateway is stateless — horizontal scaling is just more instances behind a load balancer. Redis handles shared state (rate limits, cache). For high throughput, I'd decouple audit logging via Kafka to take writes off the hot path. PostgreSQL audit logs would be range-partitioned by month for efficient archival.

21. Interview Questions & Answers (25+)

Architecture & System Design

Q1: Walk me through what happens when a request hits /v1/chat/completions.
A: 8-step pipeline: (1) Auth — SHA256 hash Bearer token, look up in api_keys, join team for config. (2) Rate limit — Redis sorted set sliding window, 60s, per-team RPM. (3) Policy — fetch team + global policies, fnmatch resource against model, priority-sorted, first match wins. (4) Cache — for temp=0, SHA256 of request params, check Redis. (5) Route — find providers for model, preference-sorted, try each with failover. (6) Cost — per-model token pricing with prefix matching. (7) Audit — insert to audit_logs. (8) Usage — upsert daily summary. Response includes gateway_metadata: cost, latency, provider, cache_hit, request_id.
If they go deeper: Ask about fail-open vs fail-closed decisions (Redis fail-open, auth fail-closed). Ask about latency overhead (~10ms for the full pipeline, negligible vs 500ms-5s LLM response).
Q2: Why FastAPI over Flask or Django?
A: (1) Async native — gateway is I/O bound waiting on LLM responses; async handles hundreds of concurrent requests in one process. Flask needs gevent/threading. (2) Pydantic integration — request validation, response serialization, settings all via Pydantic. Auto-generates OpenAPI docs. (3) Performance — FastAPI + uvicorn is one of the fastest Python frameworks, critical for a proxy adding latency to every call.
If they go deeper: "Why not Go or Rust?" — Python for rapid development and team familiarity. The gateway is I/O bound (waiting on LLM APIs), not CPU bound, so Python's GIL isn't a bottleneck. Async evloop handles concurrency fine.
Q3: How would you scale this to 10,000 RPS?
A: (1) Gateway is stateless — horizontal scale with N instances behind a load balancer. (2) Redis handles 100K+ ops/sec for rate limiting and cache. Add Redis Cluster for sharding. (3) PostgreSQL audit writes are the bottleneck at high throughput. Solution: decouple audit via Kafka, batch-insert from background workers. (4) Usage aggregation upsert needs optimistic locking or worker-based aggregation. (5) Connection pooling (PgBouncer) for DB. (6) Circuit breaker per provider to prevent cascade failures.
If they go deeper: At 10K RPS, audit_logs grows 864M rows/day. Range partition by month. Archive old partitions to S3. Read replica for dashboard queries.
Q4: Explain your rate limiting. Why sliding window over token bucket?
A: Fixed window: boundary burst problem (118 requests in 2 seconds across window boundary). Token bucket: allows controlled bursts, but LLM providers have strict rate limits — our bursts become their rate limit errors. Sliding window: exact count in any 60-second span, no bursts. Redis sorted set with timestamps as scores, ZREMRANGEBYSCORE prunes, ZCARD counts. All operations in pipeline (single round-trip). O(log N) per operation, at most 60 entries per team.
If they go deeper: "What about distributed rate limiting?" — Redis is already shared across instances. All instances write to the same sorted set. Atomic pipeline ensures consistency. For multi-region, use Redis Cluster with cross-region replication.
Q5: Why SHA256 for API keys instead of bcrypt?
A: Different threat model. Passwords: short, human-chosen, low entropy — bcrypt's intentional 100ms slowness protects against dictionary attacks. API keys: 64+ random chars, high entropy — brute force is computationally infeasible. SHA256 gives fast lookup (0.001ms vs bcrypt's 100ms) while being irreversible. Same approach as Stripe, GitHub, AWS. The key prefix (gw-) enables secret scanning in Git repos.
Q6: How does the policy engine handle conflicts?
A: Priority-based resolution. Policies have an integer priority field. Fetch all active policies (team + global), filter by resource glob match, sort by priority DESC, first match wins. "Block gpt-4*" at priority 100 beats "allow *" at priority 50. No match = default ALLOW (principle of least surprise).
If they go deeper: "What if two policies have the same priority?" — Undefined order within same priority. Best practice: use distinct priorities. Could add a tiebreaker (deny wins) as a safety measure.
Q7: What happens if Redis goes down?
A: Fail-open for rate limiting (allowed=True) and caching (cache miss). This is deliberate: the gateway's primary job is routing LLM requests. Rate limiting and caching are enhancements. Blocking all traffic because Redis is down is worse than allowing some extra traffic. The cost tracking and audit logging (PostgreSQL) still work. To improve: add in-memory fallback rate limiter per instance, and Redis Sentinel for automatic failover.
Q8: Why pre-aggregate usage instead of querying audit_logs?
A: At 1K requests/day, 3 months = 90K audit rows. "Cost by team for March" scans 30K rows. usage_summaries has ~50 rows for the same query (teams × models × 30 days). Aggregation happens on write (upsert per request) so reads are instant. Tradeoff: write amplification (extra upsert per request) but it's a simple counter increment.

Implementation & Code

Q9: Why async SQLAlchemy instead of sync?
A: Gateway is async end-to-end (FastAPI + uvicorn + httpx + redis.asyncio). Sync SQLAlchemy would block the event loop on every DB query, negating async benefits. create_async_engine with asyncpg driver gives non-blocking DB access. asyncpg is purpose-built for async PostgreSQL and significantly faster than psycopg2 in async contexts.
Q10: How does the Bedrock adapter differ from OpenAI?
A: Three differences: (1) Auth: IAM SigV4 vs Bearer token. (2) Format: Bedrock uses "Converse" API, different message structure. Adapter translates OpenAI format to Bedrock format. (3) Async: boto3 is sync, so we run it in asyncio.to_thread() to avoid blocking the event loop. Model mapping converts our names (claude-3-5-sonnet) to Bedrock IDs (anthropic.claude-3-5-sonnet-20241022-v2:0).
Q11: How would you add streaming support?
A: Three changes: (1) Gateway returns StreamingResponse (SSE) when stream=true. (2) Provider adapters request streaming and yield chunks. (3) Tricky part: cost tracking — token counts come in the final SSE chunk. Buffer the stream, forward each chunk immediately, extract usage from final chunk for audit. Cache becomes harder — need to buffer full response before caching.
Q12: What if the gateway crashes after calling LLM but before audit write?
A: Client got the response (already sent). But we miss the audit entry — cost shows in provider invoice but not our tracking. Fix: write "pending" entry before calling provider, update to "completed" after. Entries stuck as "pending" beyond timeout = lost requests. Reconcile with provider billing API monthly.
Q13: How would you implement token budget enforcement?
A: Add check between auth and rate limiting. Query usage_summaries for current month, sum total_cost for team. Cache result in Redis with 60s TTL. If spent ≥ budget, return 429. Tradeoff: team can exceed budget by up to 60s of requests during cache window. For tighter control, reduce TTL (but more DB queries).

Database & Performance

Q14: How would you partition audit_logs?
A: PostgreSQL range partitioning by created_at, monthly. Each partition is an independent table. Date-filtered queries only scan relevant partitions. Old partitions: detach, pg_dump, upload to S3, drop. Create future partitions ahead of time via cron. This is standard for append-heavy tables with time-based access.
Q15: Why UUIDs instead of auto-increment IDs?
A: (1) Don't reveal record counts or creation order to API consumers. (2) Generated client-side (no DB round-trip for ID). (3) Safe for distributed writes (no sequence conflicts across instances). Tradeoff: 16 bytes vs 4 bytes, slightly worse index locality. For our scale (not billions of rows), this is negligible.
Q16: Why PostgreSQL over MySQL or DynamoDB?
A: (1) JSONB for flexible policy conditions with native querying. (2) asyncpg is the fastest Python async driver. (3) Native UUID type. (4) Upsert with ON CONFLICT for usage aggregation. (5) Range partitioning for audit logs. DynamoDB would work for audit (append-heavy, partition by team_id + sort by timestamp) but relational queries (join team → usage → policies) are painful.

Security

Q17: What security measures are in place?
A: (1) SHA256 key hashing (shown once). (2) Admin isolation (separate key). (3) Team isolation (no cross-team access). (4) Body truncation (500 chars, reduces PII). (5) Env-based secrets. (6) Docker network isolation. (7) gw- prefix enables secret scanning. For production: add key rotation, per-key rate limits, PII redaction, and admin IP allowlist.
Q18: How would you handle prompt injection?
A: Gateway is a routing layer, not a content layer. Prompt injection prevention belongs in the application. However, the gateway can add value: (1) Content filter service in pipeline scanning for known patterns. (2) Policy conditions with content rules (deny requests containing "ignore previous instructions"). (3) Flag suspicious requests in audit logs. This would be a separate service call, similar to the policy engine.
Q19: How do you handle key rotation with zero downtime?
A: Teams can have multiple active keys simultaneously. Flow: create new key, deploy to clients, verify traffic on new key (via last_used_at), revoke old key. This is the standard approach used by AWS, Stripe, and GitHub.

Tradeoffs & Design

Q20: What tradeoffs did you make?
A: (1) Fail-open rate limiting — chose availability over strictness. (2) Synchronous audit — simpler but adds ~10ms to request path; at scale, would decouple via queue. (3) No streaming support — simplifies cost tracking but limits real-time UX. (4) Temperature=0 caching only — safe but limits cache hit rate. (5) Pre-aggregated usage — fast reads but write amplification.
Q21: What would you improve with more time?
A: (1) Streaming support (SSE). (2) Circuit breaker per provider. (3) Async audit via Kafka/Redis queue. (4) Hard budget enforcement. (5) Webhook alerts for budget thresholds and error spikes. (6) Smart model routing based on prompt complexity. (7) Multi-region with Redis Cluster.
Q22: Why did you choose an OpenAI-compatible API?
A: Teams can switch by changing one config (base_url + api_key). No code changes. The openai Python library, LangChain, LlamaIndex, and every LLM tool supports custom base URLs. This is the industry standard for LLM proxies (LiteLLM, Portkey, etc. all do this).
Q23: How does this project demonstrate your engineering skills?
A: (1) System design — multi-service architecture, clear separation. (2) API design — OpenAI-compatible, RESTful admin, proper error codes. (3) Data modeling — normalized schema, pre-aggregation, indexes. (4) Distributed systems — Redis shared state, fail-open, idempotent upserts. (5) Security — hashed keys, team isolation. (6) Testing — unit + integration, dependency injection. (7) DevOps — Docker Compose, Alembic, env-based config. (8) Full-stack — React dashboard.
Q24: How do you handle observability?
A: Three pillars: (1) Logs — structured JSON with request_id correlation via contextvars middleware. (2) Metrics — Prometheus counters/histograms for requests, latency, tokens, cost, cache hits, rate limit rejections. (3) Traces — OpenTelemetry auto-instruments FastAPI, httpx, SQLAlchemy. Each pipeline step is a span. Alerts on cost spikes (>2x 7-day average), high error rate (>5%), latency degradation (p95 > 10s).
Q25: Compare this to existing solutions like LiteLLM or Portkey.
A: Commercial gateways (Portkey, Helicone) are SaaS — you send your LLM traffic through their servers, raising data privacy concerns. LiteLLM is open-source but primarily a routing library, not a full governance platform. This gateway is self-hosted (all traffic stays in your network), includes policy engine and budget management (enterprise governance), and is purpose-built for organizations that need cost visibility across teams.

Common Mistakes to Avoid

Don't say:

  • "I just used Redis because it's fast" — explain WHY Redis vs alternatives for THIS specific use case
  • "I chose FastAPI because it's popular" — explain async I/O, Pydantic integration, performance for a proxy
  • "I cached everything" — explain temperature=0 constraint and why
  • "Security wasn't a priority" — always mention key hashing, team isolation, admin separation
  • Vague answers about scale — give specific numbers (audit rows, partition sizes, Redis ops/sec)

Do say:

  • "I chose X over Y because..." with specific tradeoffs
  • "At 10K RPS, the bottleneck would be..." with concrete numbers
  • "The fail-open design was deliberate because..." with reasoning
  • "If I had more time, I would..." showing you see the gaps
  • "In production, I would add..." showing production awareness

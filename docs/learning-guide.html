1. Project Overview & Motivation

The Problem

When an organization has 10+ teams using LLMs (OpenAI, Claude, etc.) independently:

  • No cost visibility — A $50K/month bill with no per-team breakdown
  • No access control — Any developer can call GPT-4 Turbo ($30/1M output tokens) for a task that GPT-4o-mini ($0.60/1M) could handle
  • No rate protection — One runaway loop can exhaust the org's API quota
  • Vendor lock-in — Every team imports openai directly; switching to Claude requires touching every codebase
  • No audit trail — No record of what was sent or received

The Solution

A centralized API gateway that sits between your teams and the LLM providers. Teams send requests to the gateway using the standard OpenAI-compatible API format, and the gateway handles routing, cost tracking, access control, caching, and auditing — without changing how teams write code.

3. Request Pipeline (Deep Dive)

Every request to /v1/chat/completions passes through this exact pipeline in app/routers/gateway.py:

Step 1: Authentication

Extract the Authorization: Bearer gw-xxx header. Hash the key with SHA256 and look it up in the api_keys table. If found, join with the teams table to get the team's config (budget, rate limit). Update last_used_at timestamp.

key_hash = hashlib.sha256(bearer_token.encode()).hexdigest()
api_key = await db.execute(
    select(ApiKey).where(ApiKey.key_hash == key_hash, ApiKey.is_active == True)
)

Why SHA256? We never store the raw API key. If the database is breached, the attacker gets hashes, not usable keys. SHA256 is fast enough for lookup but irreversible.

Step 2: Rate Limiting

Redis sorted set with timestamps as scores. Each request adds a member. Before adding, we remove entries older than 60 seconds (sliding window). If the set size exceeds rate_limit_rpm, deny the request.

# Redis sorted set: key="rate_limit:{team_id}"
ZREMRANGEBYSCORE key -inf (now - 60)    # Remove old entries
ZCARD key                                # Count current
ZADD key {timestamp} {request_id}        # Add new entry
EXPIRE key 120                           # Auto-cleanup

Why sorted sets? They naturally handle time-based windows. ZREMRANGEBYSCORE prunes old entries in O(log(N) + M). The 120s TTL is a safety net so abandoned keys don't consume memory forever.

Fail-open design: If Redis is unavailable, allow the request. Better to serve without rate limiting than block everyone because Redis is down.

Step 3: Policy Evaluation

Fetch all active policies for the team + global policies (team_id IS NULL). Filter to policies whose resource pattern matches the requested model using fnmatch (glob-style: gpt-4*, claude-*). Sort by priority descending. First match wins.

# Example: Block GPT-4 for intern team
Policy(
    team_id="intern-team",
    policy_type="deny",
    resource="gpt-4*",        # Matches gpt-4o, gpt-4-turbo, etc.
    priority=100,
    conditions={"max_tokens": 4096}  # Optional: only deny large requests
)

Default: ALLOW. If no policy matches, the request proceeds. This is the principle of least surprise — if you haven't configured policies, everything works.

Step 4: Cache Check

Build a deterministic cache key by hashing model + messages + temperature + top_p + max_tokens with SHA256. Only check cache for temperature=0 requests (deterministic output).

key_parts = {model, messages, temperature, top_p, max_tokens, n}
key = "cache:chat:" + SHA256(json.dumps(key_parts, sort_keys=True))

Why only temperature=0? With temperature > 0, the same prompt can produce different responses. Caching would return stale results. At temperature=0, the output is deterministic — caching is safe and saves money.

Step 5: Provider Routing

Find all providers that support the requested model. Sort by preference (Claude models prefer Bedrock, OpenAI models prefer OpenAI). Try the preferred provider; if it fails, try the next one (failover).

# Provider preference ordering:
# OpenAI models:  openai → azure → bedrock
# Claude models:  bedrock → openai → azure

for provider in providers:
    try:
        response = await provider.chat_completion(request)
        return response      # Success! Return immediately
    except Exception:
        continue              # Try next provider

Step 6: Cost Calculation

Look up the model in the pricing table. Multiply input tokens by input price and output tokens by output price. Both are per-1M-token prices.

MODEL_PRICING = {
    "gpt-4o":       (2.50, 10.00),    # $2.50/1M input, $10/1M output
    "gpt-4o-mini":  (0.15, 0.60),     # $0.15/1M input, $0.60/1M output
    "claude-3-5-sonnet": (3.00, 15.00),
    ...
}

cost = (prompt_tokens * input_price / 1M) + (completion_tokens * output_price / 1M)

Prefix matching fallback: If the exact model isn't in the table (e.g., gpt-4o-2024-08-06), we try prefix matching against gpt-4o. If nothing matches, default to conservative pricing ($10/$30 per 1M).

Step 7: Audit + Usage

Log the request to audit_logs (request_id, team, model, tokens, cost, latency, status, truncated bodies). Upsert a daily row in usage_summaries aggregating by team + model + provider + date.

Why an Adapter Pattern?

Each LLM provider has a different API format. OpenAI uses messages, Bedrock uses Converse format. The adapter pattern (abstract BaseProvider → concrete implementations) means the gateway core never knows which provider it's talking to. Adding a new provider (Google Vertex, Cohere, etc.) is just one new class.

5. Cost Tracking & Token Pricing

How Token Pricing Works

LLM APIs charge per token (roughly 0.75 words per token). Input tokens (your prompt) and output tokens (the response) have different prices. Output is always more expensive because it requires more compute (autoregressive generation).

ModelInput ($/1M tokens)Output ($/1M tokens)Example: 1K in + 500 out
gpt-4o-mini$0.15$0.60$0.00045
gpt-4o$2.50$10.00$0.0075
gpt-4$30.00$60.00$0.06
claude-3-5-sonnet$3.00$15.00$0.0105
claude-3-opus$15.00$75.00$0.0525

Key insight: GPT-4o is 16x cheaper than GPT-4 for the same quality. The gateway helps enforce cost-efficient model choices through policies.

Usage Aggregation

Instead of querying millions of audit log rows for cost reports, we maintain a usage_summaries table with pre-aggregated daily totals per team/model/provider. Each request upserts (increment) the matching row. This makes cost dashboards O(days) instead of O(requests).

6. Policy Engine

Policy Model

{
  "name": "Block expensive models for interns",
  "team_id": "intern-team-uuid",      // null = global policy
  "policy_type": "deny",               // "allow" or "deny"
  "resource": "gpt-4*",                // glob pattern matching model name
  "priority": 100,                     // higher priority wins
  "conditions": {                      // optional additional checks
    "max_tokens": 4096,                // deny if request asks for more
    "time_window": {"start_hour": 0, "end_hour": 8},  // UTC hours
    "denied_models": ["gpt-4-turbo"]   // explicit blocklist
  }
}

Evaluation Algorithm

  1. Fetch all active policies for this team + global policies
  2. Filter to policies whose resource glob-matches the model name
  3. Sort by priority descending
  4. First matching policy wins (allow or deny)
  5. If no match → default ALLOW

Why glob patterns? fnmatch gives us flexible matching. gpt-4* matches gpt-4o, gpt-4o-mini, gpt-4-turbo. claude-3-5* matches claude-3-5-sonnet, claude-3-5-haiku. This avoids maintaining explicit model lists that break when new versions are released.

7. Rate Limiting

Sliding Window Algorithm

The fixed window approach (count per minute boundary) has a burst problem: 59 requests at 0:59 + 59 requests at 1:01 = 118 requests in 2 seconds but both windows say "under limit." The sliding window fixes this by looking at the last 60 seconds from now, not from a fixed boundary.

Redis Implementation

Key:   "rate_limit:{team_id}"
Type:  Sorted Set
Score: Unix timestamp (float)
Member: "{timestamp}:{request_id}"

Operations per request:
1. ZREMRANGEBYSCORE key -inf (now - 60)   # Prune old entries
2. ZCARD key                               # Count entries in window
3. If count >= limit: DENY (remove the entry we're about to add)
4. If count < limit:  ZADD key now member  # Record this request
5. EXPIRE key 120                          # Safety TTL

Why 120s TTL? The window is 60s, so entries naturally age out. The 120s TTL is a safety net — if no new requests come, the key auto-deletes after 2 minutes instead of living forever.

Fail-Open Design

If Redis is down, the rate limiter returns allowed=True. This is a deliberate choice: it's better to allow some extra traffic than to block all traffic because the rate limiter's backing store is unavailable.

8. Response Caching

Cache Key Generation

Input:  {model, messages, temperature, top_p, max_tokens, n}
        ↓ JSON serialize (sort_keys=True, compact separators)
        ↓ SHA256 hash
Output: "cache:chat:a1b2c3d4...64 hex chars"

Why Only temperature=0?

At temperature=0, the model is deterministic — the same prompt always produces the same output. Caching is safe. At temperature > 0, the model samples randomly, so the same prompt can produce different outputs. Returning a cached response would deny the user the randomness they asked for.

Cache Invalidation

The admin API supports pattern-based invalidation via SCAN + DELETE. Example: invalidate all cache entries for GPT-4o with cache:chat:* (or a more specific pattern if we encoded the model in the key prefix — a potential improvement).

Key Design Choices

  • UUID primary keys — No sequential IDs that reveal record counts or creation order
  • key_hash not key — API keys stored as SHA256 hashes; raw key shown only on creation
  • key_prefix — First 11 chars (e.g., gw-abc1234) for identification in UI without exposing the full key
  • JSONB conditions — Flexible policy conditions without schema migration for each new condition type
  • Unique constraint on usage_summaries — (team_id, model, provider, date) enables safe upsert for aggregation
  • request_body/response_body truncated — Stored for debugging but truncated to 500 chars to avoid bloating the audit table

10. Authentication & Team Isolation

API Key Flow

1. Admin creates a team:
   POST /v1/teams {"name": "engineering"}
   → Returns team_id

2. Admin generates a key:
   POST /v1/teams/{team_id}/keys {"name": "prod-key"}
   → Returns: {"key": "gw-abc123...full64chars"}
   → Stored in DB: key_hash = SHA256("gw-abc123..."), key_prefix = "gw-abc1234"
   → The full key is shown ONCE. Never stored. Cannot be retrieved.

3. Client uses the key:
   Authorization: Bearer gw-abc123...full64chars
   → Gateway: SHA256 → look up in api_keys → join team → proceed

Why This Design?

  • Prefix gw- — Makes it easy to identify gateway keys in codebases (like sk- for OpenAI). Also enables automated secret scanning.
  • SHA256 storage — Industry standard. Even if the database leaks, the attacker can't use the hashed keys.
  • Show once — Like AWS or Stripe keys. Forces teams to store keys securely. If lost, revoke and generate a new one.

Docker Compose

    healthcheck: pg_isready
  redis:       # Redis 7 (port 6379)

Why PostgreSQL Over MySQL?

  • JSONB support — Policy conditions stored as JSONB with native querying
  • Better async support — asyncpg is the fastest Python Postgres driver
  • UUID type — Native UUID columns instead of CHAR(36)

Why Redis Over In-Memory?

  • Horizontal scaling — Multiple gateway instances share one Redis. In-memory rate limits are per-process.
  • Persistence — Cache survives gateway restarts
  • Atomic operations — Sorted set operations are atomic; no race conditions in rate limiting

14. Key Design Decisions

OpenAI-Compatible API

Decision: The gateway exposes /v1/chat/completions with the exact same request/response format as OpenAI's API.

Why: Teams can switch to the gateway by changing only the base URL and API key. No code changes. The openai Python library, LangChain, and other tools work out of the box by pointing base_url at the gateway.

Async Everything

Decision: All I/O (database, Redis, HTTP to providers) uses async/await.

Why: The gateway is I/O-bound (waiting on LLM responses). Async lets a single process handle hundreds of concurrent requests without threading overhead. FastAPI + asyncpg + httpx + redis.asyncio makes this natural.

Fail-Open for Rate Limiting and Caching

Decision: If Redis is down, rate limiting and caching are silently disabled.

Why: The gateway's primary job is routing LLM requests. Rate limiting and caching are enhancements. If Redis fails, the core functionality (routing + cost tracking + audit) should continue. The alternative (fail-closed) would block all traffic during a Redis outage.

Service Layer Separation

Decision: Business logic lives in app/services/, not in routes.

Why: Routes handle HTTP concerns (parse request, return response). Services handle business logic (calculate cost, evaluate policy). This makes services unit-testable without HTTP context and routes thin and readable.

Pre-Aggregated Usage Summaries

Decision: Maintain usage_summaries with daily aggregates instead of only querying audit_logs.

Why: After a month of use, audit_logs could have millions of rows. "Show me cost by team for March" would be a full table scan. usage_summaries has at most (teams × models × providers × days) rows — typically under 1,000 for a month. O(1,000) vs O(1,000,000).

15. Interview Questions & Answers

Architecture & Design

Q: Walk me through what happens when a request hits /v1/chat/completions.
A: The request goes through an 8-step pipeline: (1) Auth — extract Bearer token, SHA256 hash it, look up in api_keys table, join with teams to get the team's config. (2) Rate limit — check a Redis sorted set with a 60-second sliding window; if the team has exceeded their RPM, return 429. (3) Policy evaluation — fetch active policies for this team + global policies, filter by resource pattern matching the model using fnmatch glob, first match by priority wins, default allow. (4) Cache check — for temperature=0 requests, hash the request into a cache key and check Redis. (5) Provider routing — find providers supporting this model, ordered by preference, try each with failover. (6) Cost calculation — look up per-model pricing, multiply tokens by rates. (7) Audit log — insert a row with all metadata. (8) Usage aggregation — upsert a daily summary row. The response includes the LLM output plus a gateway_metadata block with cost, latency, provider, and request ID.
Q: Why did you choose FastAPI over Flask or Django?
A: Three reasons. (1) Async native — the gateway is I/O bound waiting on LLM API responses. FastAPI's built-in async support with uvicorn lets us handle hundreds of concurrent requests in a single process. Flask would need gevent or threading. (2) Pydantic integration — request validation, response serialization, and settings management all use Pydantic models. FastAPI auto-generates OpenAPI docs from these. (3) Performance — FastAPI with uvicorn is one of the fastest Python web frameworks, important for a proxy that adds latency to every LLM call.
Q: How would you scale this to handle 10,000 requests per second?
A: The current architecture handles this well because (1) the gateway is I/O bound and async, so horizontal scaling with multiple instances behind a load balancer works. (2) Redis is the shared state for rate limiting and caching — Redis handles 100K+ ops/sec easily, and we can add read replicas. (3) PostgreSQL handles the audit writes — at 10K RPS, that's 10K inserts/sec which PostgreSQL can handle, but I'd add a write buffer (queue audit logs in Redis or Kafka, batch-insert every second) to reduce DB pressure. (4) The usage aggregation upsert would need optimistic locking or a separate aggregation worker. For the provider routing, I'd add connection pooling with httpx and potentially a circuit breaker pattern to avoid cascading failures.
Q: Why did you use SHA256 for API key hashing instead of bcrypt?
A: API keys are different from passwords. Passwords are short (8-20 chars) and chosen by humans, making them vulnerable to dictionary attacks — bcrypt's intentional slowness protects against this. API keys are long (64+ chars), randomly generated, and have high entropy — a brute-force attack is computationally infeasible regardless of hash speed. SHA256 gives us fast lookup (important since we hash on every request) while being irreversible. This is the same approach used by Stripe, GitHub, and AWS for API key storage.
Q: Explain your rate limiting approach. Why sliding window over fixed window?
A: A fixed window resets at minute boundaries (e.g., 0:00, 1:00). This has a burst problem: 59 requests at 0:59 + 59 requests at 1:01 = 118 requests in 2 seconds, but both windows say "under 60." A sliding window looks at the last 60 seconds from now, which accurately represents the actual request rate. I implemented it with a Redis sorted set where each entry's score is its timestamp. ZREMRANGEBYSCORE prunes old entries, ZCARD counts current ones. All operations are O(log N). The sorted set naturally handles the time math — I don't need to track window boundaries or handle boundary-crossing logic.
Q: How does the policy engine handle conflicting policies?
A: Policies have a priority field (integer). When evaluating, I fetch all active policies for the team + global policies, filter to those whose resource pattern matches the model, sort by priority descending, and the first match wins. So if a team has a "deny gpt-4*" policy at priority 100 and a global "allow *" policy at priority 50, the deny wins. If no policy matches at all, the default is ALLOW. This is explicit over implicit — an empty policy set means "everything is allowed" which is the principle of least surprise for initial setup.

Code & Implementation

Q: Why are you using SQLAlchemy's async engine instead of sync?
A: Because the gateway is async end-to-end (FastAPI + uvicorn + httpx + redis.asyncio). If we used synchronous SQLAlchemy, every database query would block the event loop, defeating the purpose of async. With create_async_engine and AsyncSession, database I/O is non-blocking. We use asyncpg as the driver, which is purpose-built for async PostgreSQL access and is significantly faster than psycopg2 in async contexts.
Q: How do you handle the Bedrock provider differently from OpenAI?
A: Bedrock has three key differences: (1) Auth — it uses AWS IAM SigV4 signing instead of Bearer tokens, handled by boto3. (2) Message format — Bedrock uses a "Converse" API with a different message structure. The adapter translates OpenAI's [{"role": "user", "content": "..."}] format to Bedrock's format. (3) Sync client — boto3 is synchronous, so I run it in a thread executor (asyncio.to_thread) to avoid blocking the event loop. The model mapping converts our internal names (like claude-3-5-sonnet) to Bedrock's model IDs (like anthropic.claude-3-5-sonnet-20241022-v2:0).
Q: How would you implement token budget enforcement?
A: Currently, the gateway tracks usage against budgets but doesn't hard-block when a budget is exceeded — it's a monitoring feature. To enforce it, I'd add a check in the gateway pipeline between auth and rate limiting: query the usage_summaries for the current month, sum total_tokens for the team, and compare against token_budget_monthly. For performance, I'd cache this sum in Redis with a short TTL (e.g., 60 seconds) so we're not hitting Postgres on every request. The trade-off is that a team could slightly exceed their budget during the cache window, but the accuracy is sufficient for budget management.
Q: What happens if the gateway crashes mid-request after calling the LLM but before writing the audit log?
A: The user gets their response (it was already sent back before audit), but we lose that audit entry. The cost still appears in the provider's invoice but not in our tracking. To fix this at scale, I'd add a "request started" entry at the beginning (with status="pending"), then update it to "completed" or "failed" at the end. Pending entries older than the LLM timeout indicate lost requests. Alternatively, I could use a write-ahead log pattern — write the intent to Redis first (fast), then drain to Postgres asynchronously. This is a pragmatic trade-off: for an internal tool, occasional missed audit entries are acceptable; for billing, you'd want the WAL approach.

Database & Performance

Q: Why pre-aggregate usage into daily summaries instead of just querying audit_logs?
A: At 1,000 requests/day, after 3 months you have 90,000 audit rows. "Show cost by team for March" scans all 30,000 March rows. With usage_summaries, that same query reads maybe 50 rows (teams × models × providers × 30 days). The aggregation happens on write (upsert in the request pipeline) so reads are instant. The trade-off is write amplification — each request does an upsert on usage_summaries in addition to the audit insert — but that's a simple UPDATE of counters which PostgreSQL handles efficiently with the unique constraint.
Q: How would you partition the audit_logs table for long-term retention?
A: PostgreSQL native partitioning by created_at using monthly range partitions. Each month's data is in its own partition, so queries with date filters only scan relevant partitions. Old partitions can be detached and archived (moved to cold storage) without impacting active queries. I'd create partitions ahead of time with a cron job. This is a standard pattern for append-heavy tables with time-based access patterns.

Security

Q: What security measures are in place?
A: Several layers: (1) API key hashing — keys stored as SHA256, shown once on creation. (2) Admin isolation — team/policy management requires a separate admin API key, not the same key used for LLM requests. (3) Team isolation — each API key belongs to one team; there's no cross-team access. (4) Body truncation — audit logs truncate request/response bodies to 500 chars, reducing PII exposure. (5) CORS configured — the dashboard proxy prevents direct browser-to-gateway calls. (6) Environment variables — no secrets in code; all credentials via .env. (7) Docker network isolation — Postgres and Redis are only accessible within the Docker network.
Q: How would you prevent prompt injection through the gateway?
A: The gateway is a routing layer, not a content layer — it passes messages through without modification. Prompt injection prevention belongs in the application layer (before sending to the gateway) or as a dedicated middleware. However, the gateway could add value here: (1) Add a content filter service in the pipeline that scans messages for known injection patterns before routing. (2) Log and flag requests with suspicious patterns in audit logs. (3) Allow policies with content conditions (e.g., deny requests containing "ignore previous instructions"). This would be a separate service call in the pipeline, similar to how the policy engine works.

Trade-offs & Improvements

Q: What would you improve if you had more time?
A: Five things: (1) Streaming support — currently the gateway waits for the full response before returning. Real apps need SSE streaming for long completions. This requires buffering token counts for cost calculation while forwarding chunks. (2) Circuit breaker — if a provider has failed 5 times in 10 seconds, stop trying it for 30 seconds. Currently failover retries immediately. (3) Async audit — queue audit writes to Redis and batch-insert to Postgres every second, reducing DB pressure at high throughput. (4) Token budget enforcement — hard-block teams over budget, not just track. (5) Webhook notifications — alert when teams hit 80% budget or when error rates spike.
Q: If you needed to add streaming support, how would you approach it?
A: Three changes: (1) The gateway endpoint would check if stream=true in the request and return a StreamingResponse (SSE). (2) The provider adapter would request streaming from the LLM API and yield chunks. (3) The tricky part is cost tracking — in streaming mode, token counts come in the final chunk (or need to be counted client-side). I'd buffer the stream, forward each chunk to the client immediately, and collect the final usage stats from the last chunk for audit and cost. The cache becomes harder — you'd need to buffer the entire response before caching, which adds latency for cache-miss requests.
